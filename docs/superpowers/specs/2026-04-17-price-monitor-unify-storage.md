# Парсер цен — единое хранилище и batch-запрос для списка

**Дата:** 2026-04-17
**Статус:** approved

## Проблема

Сейчас в парсер-сервисе две независимые таблицы:

- `offers` (v2) — плоская таблица прайсов. Ночной скрипт `runner.py` пишет только сюда. Оттуда читают `/offers` (для индикатора в списке товаров) и `/notifications` (для колокольчика).
- `site_results` (v3) — одна запись на пару «товар × сайт» со статусом (OFFERS / OUT_OF_STOCK / NOT_FOUND / ERROR / NOT_CONFIGURED), списком оффераов и метаданными. Пишется только при ручном нажатии «Обновить» в карточке товара. Оттуда читает `/site-results` (для таблицы 5 сайтов в карточке).

**Последствия:**

1. Карточка товара не показывает результаты ночного парсинга — она смотрит в `site_results`, а ночь пишет в `offers`. Пока не нажмёшь «Обновить» в карточке, там пусто.
2. Данные могут расходиться между колокольчиком/индикатором и карточкой, потому что источники разные.
3. Дублирующая логика в коде (две системы хранения, две записи).

Отдельная проблема: список товаров в админке делает по одному запросу на каждую строку (`/offers?article=...`), получая одновременно 20+ сетевых походов за уже кешированными данными. Тормозит открытие раздела.

## Цель

1. **Одна таблица = один источник правды.** Все три вида (колокольчик, индикатор в списке, таблица в карточке) читают из `site_results`.
2. **Ночной скрипт пишет в `site_results`** той же логикой, что и ручное «Обновить».
3. **Batch-запрос `/bulk`** для списка товаров — один запрос возвращает рыночные сводки для всех артикулов страницы.

## Архитектура

### Хранилище (парсер, SQLite)

- Остаётся только `site_results`.
- Таблица `offers` удаляется (дамп сохраняется в `offers_backup_YYYY-MM-DD.sql` на случай отката).
- `brand_aliases`, `sources` — не трогаем.

### API парсера (FastAPI)

- `GET /site-results?article=&brand=` — без изменений, читает `site_results`.
- `GET /offers?article=&brand=` — теперь **вычисляет** мин/медиану/макс из `site_results` (агрегируя все офферы по всем сайтам для этой пары). Формат ответа тот же — ломать фронт не нужно.
- `GET /notifications` — так же из `site_results`: агрегирует, считает медиану, сравнивает с `your_price` из `articles.txt`, раскладывает по зонам green/yellow/red. Формат ответа тот же.
- `POST /parse-v3` — без изменений.
- **Новый:** `POST /bulk` с телом `{ items: [{article, brand}, ...] }` → возвращает `{ results: [{article, brand, min_price, max_price, median_price, sites_count, offers_count, scraped_at}] }`. Для списка товаров.

### Ночной runner

- `runner.py` переписывается так, чтобы для каждой пары `(article, brand)` вызывать тот же путь, что и `/parse-v3`: прогоняет все 5 скрейперов, пишет результаты в `site_results` через `insert_site_result`.
- Удаляется вызов `clear_offers` + `insert_offers`.
- Логика задержек и параллелизма (DELAY_MIN/MAX, WORKERS) — та же.

### Фронтенд (admin)

- `ProductList` → вместо `fetchMarketData` на каждую строку использует новый `fetchMarketDataBulk(articles)`, вызывающий `/api/price-monitor/bulk`. Один запрос на страницу.
- Новый прокси-роут `/api/price-monitor/bulk/route.ts` — просто пересылает на `PARSER_URL/bulk` с X-API-Token.
- Карточка товара (MarketPriceWidget) — не трогаем.
- Колокольчик — не трогаем.

## Миграция данных

Одноразовый шаг при деплое:
1. Дамп `offers` в файл бэкапа.
2. Дроп таблицы.
3. Запуск `runner.py` руками сразу после деплоя → заполняет `site_results` свежими данными за ~10–15 мин.

## Риски и проверки

**Риски:**
- Формат `/offers` и `/notifications` снаружи должен остаться прежним, иначе сломается существующий фронт.
- Ночной парсинг теперь пишет 5 строк (по сайтам) вместо N офферов — объём данных сопоставим.

**Приёмка:**
- [ ] Карточка товара показывает цены без нажатия «Обновить» — берёт данные, записанные ночью.
- [ ] Список товаров открывается одним сетевым запросом к `/bulk` (проверить в DevTools → Network).
- [ ] Колокольчик показывает те же отклонения, что и до переделки.
- [ ] Таблица в карточке совпадает по цене с индикатором в списке (не расходятся).

## Что НЕ входит

Из аудита отложены на отдельные спеки:
- Массовые операции (чекбоксы + bulk edit) — P1 #4.
- Inline-редактирование цены в списке — P1 #5.
- Сохранение фильтров в URL — P1 #2.
- Возврат из карточки в ту же позицию списка — P1 #3.
- Баннер «Сохранено» — P1 #6.
- Undo для удаления — P1 #7.
- Сортировка, фильтры «выше/ниже рынка», счётчик результатов — P2.
