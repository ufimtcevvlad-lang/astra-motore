# Дизайн: Парсер цен v2 — 6 источников

## Цель

Узнать реальную рыночную цену на каждую запчасть в каталоге, собирая предложения с 6 площадок. Владелец видит ценовой коридор и принимает решение — не продешевить и не завысить.

Система не навязывает формулу. Показывает все данные — человек решает сам.

## Источники

### 1. exist.ru — HTML с embedded JSON

- **URL:** `https://www.exist.ru/Price/?pcode={article}`
- **Метод:** HTTP GET → парсим `_data` JSON из `<script>` тега (bracket-counting)
- **Данные:** список предложений с ценой (int ₽), сроком доставки (minutes), продавцом
- **Фильтр:** только предложения с доставкой <= 7 дней (10080 минут)
- **Авторизация:** cookie-сессия (логин/пароль). Без авторизации — оптовые цены (занижены)
- **Поиск бренда:** поле `CatalogName` в массиве `_data`

### 2. emex.ru — SSR страница товара (Next.js)

- **URL:** `https://emex.ru/products/{article}/{brand_lower}/{locationId}`
- **locationId:** 39915 (Екатеринбург)
- **Метод:** HTTP GET → извлечь `initialState` JSON из HTML
- **Данные:** предложения с ценой, рейтингом продавца, наличием, сроком доставки
- **Авторизация:** cookie-сессия (логин/пароль). С авторизацией — реальные розничные цены
- **Важно:** старый метод через `/api/search/search` давал bestPrice-заманушку. Не использовать.

### 3. armtek.ru — исследовать API/HTML

- **Известно:** REST API существует (`ws.armtek.ru/api/ws_search/search?format=json`), Basic Auth
- **Данные:** цена, наличие, сроки доставки
- **Авторизация:** Basic Auth (логин/пароль)
- **Статус:** API не поддался в ручном тестировании. При реализации — исследовать глубже, при неудаче парсить HTML

### 4. part-kom.ru — SSR JSON (Next.js)

- **URL:** `https://part-kom.ru/search?search={article}`
- **Метод:** HTTP GET → парсить JSON из `__next_f` SSR payload
- **Данные:** цена, наличие, бренд
- **Авторизация:** cookie-сессия (логин/пароль)

### 5. vdopel.ru — JSON API (без авторизации)

- **Шаг 1:** `GET /detailSearchNew/default/getSkladList/?search_phrase={article}&time_start=0`
  - Возвращает: `{"sklads_count":1,"sklads":["local_my"]}`
- **Шаг 2:** `GET /detailSearchNew/default/getProductList/?search_phrase={article}&search_sklad=local_my&time_start=0&search_brand=`
  - Возвращает: цена, наличие (кол-во), 4 ценовых уровня (розница/опт), аналоги
- **Авторизация:** не нужна
- **Специализация:** Opel — прямо в тему каталога

### 6. plentycar.ru — CSV прайс-лист

- **Метод:** скачать CSV/XLSX файл с ценами (обновляется 2 раза в день)
- **Данные:** артикул, бренд, цена, наличие, склад (Москва, Екатеринбург)
- **Авторизация:** разобраться при реализации
- **Особенность:** не парсинг сайта, а загрузка файла. Поиск артикула по скачанному прайсу

## База данных (SQLite)

Хранится на VPS рядом с парсером. Только актуальные данные — история не нужна. Перед каждым прогоном таблица offers очищается и заполняется заново.

```sql
CREATE TABLE sources (
    id INTEGER PRIMARY KEY,
    site TEXT UNIQUE NOT NULL,
    enabled INTEGER DEFAULT 1,
    auth_config TEXT,
    last_success_at TEXT,
    last_error TEXT
);

CREATE TABLE offers (
    id INTEGER PRIMARY KEY,
    article TEXT NOT NULL,
    brand TEXT NOT NULL,
    site TEXT NOT NULL,
    price REAL NOT NULL,
    delivery_days INTEGER,
    in_stock INTEGER,
    seller_name TEXT,
    scraped_at TEXT NOT NULL
);

CREATE INDEX idx_offers_lookup ON offers(article, brand);
```

Перед каждым прогоном: `DELETE FROM offers;` — затем вставляем свежие.

## FastAPI — эндпоинты

```
POST /parse?article=XXX&brand=YYY
  → парсит одну запчасть по всем сайтам, возвращает список предложений

POST /parse-all
  → прогоняет весь каталог (articles.txt), перезаписывает offers

GET  /offers?article=XXX&brand=YYY
  → отдаёт сохранённые предложения из БД (быстро, без парсинга)

GET  /status
  → статус источников: какой работает, когда последний успех/ошибка
```

## Ночной прогон (cron, 03:00)

1. Загружает `articles.txt` (артикул|бренд|наша_цена)
2. По каждому артикулу — запрос ко всем 6 сайтам параллельно (с задержками между запросами)
3. `DELETE FROM offers` → вставить свежие
4. Результат: таблица `offers` содержит актуальные предложения рынка

## Единый формат предложения

Каждый скрапер возвращает список объектов:

```python
@dataclass
class Offer:
    article: str
    brand: str
    site: str           # 'exist.ru', 'emex.ru', ...
    price: float        # в рублях
    delivery_days: int | None    # null если неизвестно
    in_stock: bool | None        # null если неизвестно
    seller_name: str | None      # null если сайт не отдаёт
```

Сайты с богатыми данными (exist, emex, vdopel) — заполняют всё. Сайты с бедными данными (plentycar — только цена) — остальные поля null.

## Отображение в админке

### Карточка товара

```
Твоя цена: 450₽

Рынок (6 источников, обновлено 16.04 в 03:00):
  exist.ru      332₽  (в наличии, 2 дня)
  emex.ru       309₽  (в наличии, 7 дней)
  armtek.ru     380₽  (в наличии)
  part-kom.ru   509₽  (в наличии)
  vdopel.ru     330₽  (в наличии, 19 шт)
  plentycar.ru  410₽  (в наличии)

  Мин: 309₽ | Медиана: 395₽ | Макс: 509₽
```

### Цветовая индикация

- Красный — твоя цена выше максимальной рыночной (продаёшь дорого)
- Зелёный — твоя цена в коридоре мин–макс (в рынке)
- Жёлтый — твоя цена ниже минимальной (возможно продешевил)

### Список товаров

Колонка "Рынок" — мини-индикатор (красный/зелёный/жёлтый) + диапазон "309–509₽".

### Уведомления

- Колокольчик в шапке админки — кол-во товаров в красной/жёлтой зоне
- Утренний блок на главной админки — сводка после ночного прогона: сколько спарсили, сколько красных/жёлтых/зелёных

Зелёная зона уведомлений не генерирует — только красная и жёлтая.

## Порядок реализации

1. Переписать exist.ru — добавить авторизацию (cookie-сессия)
2. Переписать emex.ru — парсить страницу товара вместо API, добавить авторизацию
3. Новый: vdopel.ru — JSON API, самый простой
4. Новый: part-kom.ru — SSR JSON, с авторизацией
5. Исследовать armtek.ru — разобраться с API или парсить HTML
6. Новый: plentycar.ru — CSV прайс
7. Обновить БД — таблицы sources + offers
8. Обновить API — эндпоинты /offers, /status, /parse-all
9. Настроить cron — ночной прогон в 03:00
10. Обновить админку — таблица предложений + цветовой индикатор

## Что система НЕ делает

- Не ставит цену автоматически — только показывает рынок
- Не хранит историю — только актуальный срез
- Не парсит аналоги — только точное совпадение артикул+бренд
- Не использует Playwright/headless — только HTTP-запросы

## Стек технологий

- Python 3.11+, FastAPI, httpx (async), BeautifulSoup4
- SQLite (на VPS)
- Cron (systemd timer или crontab)
- Next.js админка (существующая) — новые компоненты для отображения
