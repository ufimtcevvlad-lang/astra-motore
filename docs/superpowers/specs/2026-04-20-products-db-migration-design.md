# Переход публичного сайта на чтение товаров из БД

**Дата:** 2026-04-20
**Worktree:** hungry-clarke-699fd6

## Проблема

Сейчас публичные страницы магазина (главная, каталог, `/zapchasti-*`, карточки, корзина, поиск) читают список товаров из статического файла `src/app/data/products.ts` — 3378 строк, 121 товар захардкожен. Админка и импорт из Excel пишут в SQLite, но публичный сайт игнорирует БД. В результате любой импорт через админку невидим для посетителей.

Наследие начальной генерации проекта в Cursor: сайт стартовал со статического массива, позже прикрутили админку на БД, но публичный слой не переключили.

## Цель

Публичный сайт читает товары из SQLite. Любое добавление/изменение через админку мгновенно отражается на сайте после ревалидации. Статический файл `products.ts` удаляется.

## Архитектура

**Источник правды:** таблица `products` в SQLite (`data/shop.db`, доступ через Drizzle).

**Расширение схемы:** в `products` добавляются два поля:
- `slug TEXT UNIQUE NOT NULL` — канонический URL-сегмент карточки
- `description TEXT` — описание (генерируется авто-генератором при создании, редактируется вручную)

Поля из старой статики, которые НЕ переносим: `specs`, `features`, `compatibility`, `rating`, `reviews`. Обоснование — они есть только у 121 захардкоженного товара, у новых импортов их не будет и появление реальных данных не планируется. Хранить фейковые рейтинги и отзывы для части каталога = неконсистентный UX.

**Фото товаров:** поля `image` в БД нет. Галерея читается из директории `public/uploads/products/<sku>/` (соглашение, см. memory «Фото товаров»). Парсер фото раскладывает файлы по папкам по артикулу.

**Хелпер доступа к данным:** `src/app/lib/products-db.ts` экспортирует функции с сигнатурами, аналогичными тому, что сейчас даёт `data/products.ts`:
- `getAllProducts(): Product[]`
- `getProductBySlug(slug: string): Product | null`
- `getProductBySku(sku: string): Product | null`
- `getProductsByCategory(slug: string): Product[]`
- `getProductsByBrand(brand: string): Product[]` — для страниц `/zapchasti-opel` и т.п.
- `searchProducts(query: string): Product[]`

Хелпер делает SELECT, маппит row → Product-объект (с галереей через `getProductImages(sku)`). Тип `Product` остаётся прежний, чтобы минимизировать изменения в компонентах.

**Хелпер галереи:** `src/app/lib/product-images.ts` экспортирует `getProductImages(sku: string): string[]` — читает `fs.readdirSync("public/uploads/products/<sku>")`, возвращает пути вида `/uploads/products/<sku>/<file>`. Если папки нет или пуста — возвращает `["/placeholder-product.svg"]` (существующий placeholder).

**Кеширование:** SSG + `revalidatePath()`. Публичные страницы остаются статическими, но при мутациях (админка: создание/обновление/удаление товара или массовый импорт) мы триггерим `revalidatePath` для соответствующих маршрутов:
- `/` (главная)
- `/catalog`
- `/zapchasti-gm`, `/zapchasti-opel`, `/zapchasti-chevrolet`
- `/product/<slug>`

**Триггер папки фото:** при создании товара (через админку или confirm-импорт) вызывается `ensureProductDir(sku)` — создаёт пустую `public/uploads/products/<sku>/` если не существует.

## Миграция 121 товара

Одноразовый скрипт `scripts/seed-products-from-static.ts`:
1. Импортирует массив из `data/products.ts`.
2. Для каждого товара:
   - Генерирует slug через существующую `getProductSlug(p)`.
   - Вставляет в БД с `externalId = "static-" + old.id` (чтобы не конфликтовать с `import-*` от Excel-импорта).
   - Description — берёт из старого `p.description` (не авто-генерирует, сохраняем что было).
   - categoryId — маппит `p.category` (slug) через таблицу `categories`.
   - `mkdir public/uploads/products/<sku>` (папки пустые, наполнятся парсером фото позже).
3. Пропускает, если товар с таким SKU уже есть в БД (идемпотентность).

Запуск один раз при первом деплое, перед переключением кода.

## Порядок работ (без даунтайма, безопасный)

1. **Миграция схемы** — новая Drizzle-миграция добавляет `slug`, `description`. Индекс uniq на slug.
2. **Сид-скрипт** — `scripts/seed-products-from-static.ts`, можно запустить локально на прод-БД через SSH до деплоя кода.
3. **Хелпер `products-db.ts`** — реализует публичный API поверх Drizzle.
4. **Хелпер `product-images.ts`** — читает директорию фото.
5. **Переключение 12 публичных файлов** по одному, с локальной проверкой каждого в браузере:
   - `src/app/zapchasti-opel/page.tsx`
   - `src/app/zapchasti-gm/page.tsx`
   - `src/app/zapchasti-chevrolet/page.tsx`
   - `src/app/components/HomeFeatured.tsx`
   - `src/app/components/ProductCatalog.tsx`
   - `src/app/components/RecentlyViewed.tsx`
   - `src/app/components/CartContext.tsx`
   - `src/app/cart/page.tsx`
   - `src/app/favorites/FavoritesPageContent.tsx`
   - `src/app/lib/catalog-search.ts`
   - `src/app/lib/product-analogs.ts`
   - `src/app/lib/product-slug.ts` — читает массив на стадии модуля, строит карту slug ↔ product. Переписать на вычисление slug только по одному объекту (ленивое), без общей карты.
   - `src/app/lib/product-description-gen.ts` — только `type Product`, массив не читает. Нужно заменить тип-импорт на новый тип из `products-db.ts` (или оставить общий `types/product.ts`).
6. **Админка — мутации + ревалидация:**
   - Создание/обновление/удаление товара → `revalidatePath`.
   - Подтверждение Excel-импорта → `revalidatePath` + создание папок фото по каждому SKU.
7. **Локальная проверка** всех публичных сценариев (см. раздел «Тестирование»).
8. **Удаление `src/app/data/products.ts`** — TS-компилятор покажет оставшиеся импорты.
9. **Деплой:**
   - Коммит и push ветки.
   - На проде: `git pull && npm ci && npm run migrate && npm run seed:static && npm run build && pm2 restart astra-motors`.
   - Сид идемпотентен: если прокатился ранее — пропустит.

## Тестирование (локальный чек-лист перед деплоем)

Все страницы должны выглядеть и работать как сейчас:

- [ ] `/` — HomeFeatured показывает товары.
- [ ] `/catalog` — полный список, фильтры, пагинация.
- [ ] `/zapchasti-opel`, `/zapchasti-gm`, `/zapchasti-chevrolet` — правильные подборки по бренду.
- [ ] `/product/<slug>` — карточка старого товара открывается, описание то же, галерея placeholder (фото пока нет в папках).
- [ ] Поиск `/api/search?q=...` — находит по name/brand/sku/car.
- [ ] Корзина — добавляем товар, закрываем вкладку, возвращаемся — корзина сохранилась.
- [ ] Избранное — аналогично.
- [ ] Админка: создаём новый товар → он появляется на `/catalog` после ревалидации (ручной `revalidatePath` или перезагрузки страницы).
- [ ] Админка: Excel-импорт → новые товары видны на публичных страницах.

## Риски и мигация

| Риск | Мигация |
|---|---|
| URL-слаги изменятся — Google потеряет индексацию. | Slug в сиде генерим той же `getProductSlug()`, что и в рантайме. Проверяем на локалке: для каждого из 121 товара новый slug === старый. |
| После удаления `products.ts` забытый импорт сломает билд. | TS-компиляция после удаления покажет все ссылки. Исправляем перед коммитом. |
| Семантика `Product` в DB отличается от TS — компоненты сломаются. | Хелпер `products-db.ts` маппит row → Product с тем же типом. Типы не меняем. |
| Сид запустится дважды на проде, продублирует записи. | Идемпотентность: проверка на существование SKU перед INSERT. |
| Ревалидация забыта в каком-то endpoint админки — изменения невидимы. | Добавляем во все мутации products. Список: create, update, delete, bulk, import/confirm, [id]/quick, [id]/duplicate. |
| Пути к фото `/uploads/products/<sku>/` попадают в .gitignore и теряются при деплое. | `public/uploads/` уже в .gitignore, фото льются парсером отдельно на прод. Папки сами не нужны в репо — создаются при создании товара. |

## Границы задачи (YAGNI)

Не входит:
- Редактор описаний в админке с rich-text (существующий `<textarea>` остаётся).
- Миграция реальных фото — парсер фото существует отдельно, льёт в `public/uploads/products/<sku>/` в своё время.
- Изменение UX каталога, добавление новых фильтров.
- Рефакторинг логики формирования slug.
- Удаление `car-models.ts`, `catalog-sections.ts`, `non-gm-markers.ts` — остаются как data-файлы (они не источник товаров).
