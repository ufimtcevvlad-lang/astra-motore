# Каталог и пост-обработка импорта из 1С

**Статус:** черновик на согласование
**Дата:** 2026-04-19
**Контекст:** магазин GM-запчастей (Opel / Chevrolet / Cadillac), ассортимент 5700+ позиций после фильтрации 1С-прайса. Текущий каталог из 7 групп / 13 разделов слишком мал для такого объёма. Импорт через Excel в админке уже есть ([`src/app/api/admin/products/import/route.ts`](../../../src/app/api/admin/products/import/route.ts)), но не дописывает марку авто, не присваивает категорию и не генерирует SEO-название.

## 1. Цели

1. Утвердить структуру каталога под 5700+ реальных позиций из 1С-прайса.
2. При импорте Excel автоматически формировать SEO-наименование (марка авто + артикул в конце), присваивать категорию, определять марку авто.
3. Отсекать на импорте позиции не-GM (VAG, Hyundai, Toyota и т.п.) и жидкости/химию (под «Честным знаком» не продаём).
4. Админу — превью с правками до записи в БД и batch-инструмент для разметки хвоста.

## 2. Структура каталога

### 2.1 Иерархия

Два уровня в БД: **группа → раздел**. Товар ссылается на раздел (`categoryId`). Третий уровень (по модели авто) — виртуальный: фильтр по модели в пределах раздела + SEO-хабы.

### 2.3 SEO-хабы по моделям (виртуальные страницы)

- URL: `/zapchasti/{make}-{model}/` (например `/zapchasti/opel-astra-h/`).
- Страница динамически собирает блоки «в этом разделе для {модель}» из товаров, где поле `car` содержит эту модель.
- В БД новые сущности не заводим.
- Список моделей берётся из справочника [`src/app/data/car-models.ts`](../../../src/app/data/car-models.ts).

### 2.4 Фильтр по модели внутри раздела

- URL: `/catalog/{section-slug}?model={model-slug}` (например `/catalog/tormoznye-kolodki?model=astra-h`).
- Фильтрация по подстроке в `car`. Без query-параметра — все товары раздела.

## 3. Пайплайн пост-обработки импорта

### 3.1 Последовательность шагов

```
Excel из 1С (C=name, D=sku, E=brand, F=price)
        ↓
[1] Парсинг (существующий код)
        ↓
[2] Фильтр не-GM: если name или brand содержит не-GM марку → в отбракованные
        ↓
[3] Фильтр химии/жидкостей: если name содержит «масло», «жидкость», «антифриз»,
    «шампунь», «грунт», «полироль», «очиститель», «смазка», маркеры объёма
    (мл, л, кг) совместно с брендами химии → в отбракованные
        ↓
[4] Определение марки авто: по справочнику car-models.ts → make + canonical model
        ↓
[5] Переписывание name:
    - вставляем марку перед моделью (если марки в name ещё нет)
    - нормализуем дефисы в моделях (Astra-H → Astra H)
    - дописываем ` | арт. {sku}` в конец
        ↓
[6] Определение categoryId по правилам category-rules.ts (~100-150 правил)
    - если нет совпадения → null (попадает в очередь «Без категории»)
        ↓
[7] Превью в админке: каждая строка с пометками
    - зелёная: категория определена
    - жёлтая: «требует ручной разметки» (categoryId = null)
    - красная: отбраковано (не-GM или химия)
    - админ может редактировать полную строку перед подтверждением
        ↓
[8] Подтверждение → запись в БД (только зелёные и жёлтые, красные пропускаются)
```

### 3.2 Правила переписывания `name`

**Формула:** `{name_с_маркой} | арт. {sku}`

**Алгоритм вставки марки:**
1. Нормализовать name — заменить дефисы в моделях на пробелы (`Astra-H` → `Astra H`, исправить опечатки по `TYPO_FIXES`).
2. Проверить, есть ли в name уже марка (`MAKE_TOKENS_IN_NAME` из `car-models.ts`). Если есть — пропустить шаг 3.
3. Найти в name первое вхождение любой модели из справочника. Вставить марку перед ней.
4. Обработать мульти-модельные кейсы (`Cruze/Astra J`):
   - если обе модели одной марки → `Opel Astra H/J`
   - если разных марок → `Opel Astra J/Chevrolet Cruze`
5. В самом конце дописать ` | арт. {sku}`.

**Примеры:**

| Name из 1С | Name в БД |
|---|---|
| `Фильтр масляный HENGST Astra-H/J A16XER` | `Фильтр масляный HENGST Opel Astra H/J A16XER | арт. E611HD442` |
| `Бачок расширительный без датчика Cruze/Astra J` | `Бачок расширительный без датчика Chevrolet Cruze/Opel Astra J | арт. 13465094` |
| `Колодки тормозные барабанные MILES Opel Corsa D` | `Колодки тормозные барабанные MILES Opel Corsa D | арт. E210207` (марка уже есть, не дублируем) |
| `Свеча зажигания BOSCH Z16XEP/XER` | `Свеча зажигания BOSCH Z16XEP/XER | арт. 0242229699` (только коды двигателей — марку не подставляем) |

### 3.3 Правила определения категории

**Файл:** `src/app/data/category-rules.ts`.

**Формат:**
```ts
export interface CategoryRule {
  sectionSlug: string;
  keywords: string[];  // проверяются по name.toLowerCase(), первое совпадение побеждает
}

export const CATEGORY_RULES: CategoryRule[] = [
  { sectionSlug: "tormoznye-kolodki", keywords: ["колодки тормозн"] },
  { sectionSlug: "tormoznye-diski",   keywords: ["диск тормозн"] },
  // ...~100-150 правил
];
```

**Порядок:** от специфичных к общим. `колодки тормозные задние` до `колодки`, `фильтр масляный` до `фильтр`.

**Ожидаемое покрытие:** ~88-90% на прогоне по 5700-позиционному прайсу (см. анализ в истории переписки).

## 4. UI превью импорта

Существующий компонент [`src/app/admin/components/ExcelImport.tsx`](../../../src/app/admin/components/ExcelImport.tsx) расширяется:

- Вместо двух секций (Новые / Совпадения) — три: **Новые**, **Совпадения**, **Отброшено**.
- Для каждой «Новой» позиции видны: финальное `name` (с марками и артикулом), предполагаемая `category` (название раздела), предполагаемая `car`.
- У каждой позиции — редактируемые поля (имя, бренд, цена, категория — селект, марка авто — селект).
- Жёлтая подсветка строк с `categoryId = null`.
- «Отброшено» сворачивается, показывает причину (не-GM / химия).

## 5. Batch-разметка хвоста в админке

На странице списка товаров в админке (`src/app/admin/(app)/products/page.tsx`):

- Фильтр «Без категории» (`categoryId IS NULL`).
- Множественный выбор (чекбоксы + «выделить всю страницу»).
- Кнопка «Назначить категорию» → модалка с селектом → подтверждение.
- Обычный PATCH в бэкенде, без новой логики.

## 6. Изменения в схеме БД

Нужна таблица `categories` с группой и разделом. Сейчас в коде `CATALOG_SECTIONS` — хардкод в [`src/app/data/catalog-sections.ts`](../../../src/app/data/catalog-sections.ts), товары ссылаются на раздел по строке `category`. Два пути:

**A. Оставить хардкод.** Обновить `CATALOG_SECTIONS` до 13 групп / ~50 разделов. `products.categoryId` → `text categorySlug`. Проще, но админить категории можно только через код.

**B. Завести реальные таблицы `catalog_groups` и `catalog_sections`.** В админке — CRUD по ним. Гибче, но больше работы.

**Рекомендация:** стартовать с **A** (хардкод), чтобы выкатить быстро. Переход на **B** — отдельный спек, когда понадобится добавлять разделы без деплоя.

## 7. Что остаётся вне скоупа этого спека

- Реализация UI хабов `/zapchasti/{make}-{model}/` — отдельная задача после утверждения каталога.
- Переопределение URL-структуры товаров `/product/{slug}` — пока остаётся как есть, slug генерируется на лету.
- Миграция существующих 121 товара из БД в новую структуру категорий — отдельная задача (нужно решить, запускать ли эти же правила категоризации на существующих `name`).
- Расширение формата Excel (дополнительные колонки) — пока работаем с текущими C/D/E/F.
- Полный редизайн страницы каталога — текущая UI-структура не меняется.

## 8. Риски и открытые вопросы

**Риски:**

- Покрытие автомата может оказаться ниже 88% на реальном потоке, если 1С-прайс будет содержать новые типы. Митигация: лог «непокрытых» типов накапливается, админ раз в месяц дополняет `CATEGORY_RULES`.
- Мульти-модельные кейсы (`Cruze/Astra J`) редкие, но при ошибке выдают неграмотное название. Митигация: админ видит превью и правит руками.
- Фильтр не-GM может ошибочно отбросить GM-позицию, если в названии случайно есть токен другой марки (очень редкий кейс). Митигация: такие позиции видны в «Отброшено», админ может вернуть вручную.

**Открытые (решим в процессе реализации):**

- Какие именно slug у разделов? (Ниже пример, финально утвердим при написании кода.)
- Нужна ли сортировка разделов внутри группы (алфавитная / по частотности)?

## 9. Метрики успеха

- После импорта 5700 товаров: ≥ 88% с `categoryId != null`, ≤ 7% отброшено как не-GM/химия, оставшиеся ≤ 5% в очереди «Без категории».
- Переписанное `name` содержит марку авто в ≥ 90% случаев, где модель присутствует в справочнике.
- Админ импортирует 100 товаров за ≤ 5 минут (включая проверку превью).
