# Справка: расширенные правила (catalog-product-media)

## Два слоя

- **CSS (Next/Image):** как картинка ведёт себя в UI — основной способ в проекте.
- **Пакетная обработка (обязательна при загрузке фото в репозиторий):**
  - **`npm run catalog:normalize-images:recompress-png`** — основной рабочий вариант для агента: `scripts/normalize-catalog-images.mjs` с флагом **`--recompress-png`**. Делает EXIF-поворот, вписывает кадр в **1600×1600** без апскейла, пережимает **PNG** даже если стороны уже **<1600px** (типичные фото с телефона иначе бы не трогались). JPEG тоже проходят через пайплайн. Если выходной файл крупнее исходника **>5%**, исходник не заменяют.
  - **`npm run catalog:normalize-images`** — без `--recompress-png`: PNG/WebP **без** даунскейла **не** пересохраняются (защита от раздувания файла при слепом перекодировании). Использовать только если осознанно не нужно трогать мелкие PNG; для новых загрузок в каталог по скиллу — **не этот** вариант.

Для очень больших каталогов (~25k+) позже возможен отдельный WebP/CDN-пайплайн; до тех пор держим диск и git через обязательный прогон **`catalog:normalize-images:recompress-png`** после добавления файлов.

## Next.js

- В режиме `next start` компонент `Image` отдаёт оптимизированные форматы на лету — сырые PNG в `public/` могут оставаться как есть при умеренном размере.

## Техника файлов

- Экспорт в **sRGB**; при необходимости сброс чувствительных EXIF.
- **Прозрачный PNG:** на белой карточке проверять кайму; при артефактах — холст с белым фоном.

## Проверки

- Пути из `products.ts` существуют на диске.
- Один файл на два SKU — только осознанно.
- Заглушка `_pending.jpg` — единый стиль.

## SEO / фиды

- При появлении Merchant / маркетплейсов — отдельные требования к размеру и фону; не смешивать с вёрсткой магазина без явной задачи.

## Будущее

- 360°, видео: отдельное соглашение (постер, lazy, без автозвука).
