# Справка: расширенные правила (catalog-product-media)

## Два слоя

- **CSS (Next/Image):** как картинка ведёт себя в UI — основной способ в проекте.
- **Пакетная обработка (обязательна при загрузке фото в репозиторий):**
  - **`npm run catalog:normalize-images`** — основной режим: `scripts/normalize-catalog-images.mjs --to-webp`. Поворот EXIF, максимальная сторона **1600px**, прозрачность PNG → flatten на **#ffffff**, выход **WebP** (quality **86**, при сильном раздувании второй проход **80**). Файлы в каталоге становятся **`.webp`**; пути в **`products.ts`** и **`commons-sources.json`** обновляются автоматически.
  - **`npm run catalog:normalize-images:legacy`** — прежняя логика «не трогать мелкие PNG без флага».
  - **`npm run catalog:normalize-images:recompress-png`** — legacy + `--recompress-png` (пережатие PNG без перехода на WebP).

Проверка без изменений: `node scripts/normalize-catalog-images.mjs --to-webp --dry-run`.

## Next.js

- Статические **`.webp`** в `public/` отлично отдаются; `Image` дополнительно может отдавать AVIF/WebP варианты в рантайме.

## Техника файлов

- Экспорт в **sRGB**; при необходимости сброс чувствительных EXIF.
- **Прозрачный PNG:** скрипт кладёт на белый фон — на карточке с белым фотобоксом визуально согласовано.

## Проверки

- Пути из `products.ts` существуют на диске (после `--to-webp` — с расширением `.webp`).
- Один файл на два SKU — только осознанно.
- Заглушка `_pending.jpg` — не трогается скриптом.

## SEO / фиды

- При появлении Merchant / маркетплейсов — отдельные требования к размеру и фону; не смешивать с вёрсткой магазина без явной задачи.

## Будущее

- 360°, видео: отдельное соглашение (постер, lazy, без автозвука).
